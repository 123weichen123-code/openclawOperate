# STS2 Agent 实现方案

## 目标

目标不是只做一个“会点按钮”的脚本，而是做一个可以稳定自主游玩《杀戮尖塔2》（Slay the Spire 2）的 agent 系统，并且在架构上达到或超过现有 STS1 项目的工程化水平。

最终希望达到的能力：
- 自动开始新局
- 自动进行战斗决策
- 自动选择奖励、路线、商店购买、事件分支
- 支持规则引擎和 agent/LLM 双模式
- 可解释、可测试、可扩展

---

## 现有 GitHub 项目分析

## 1. `ForgottenArbiter/spirecomm`
- 价值定位：**通信层模板**
- 核心能力：把 Slay the Spire 游戏状态通过 Communication Mod 暴露给外部进程
- 借鉴点：
  - 游戏内外通信桥接
  - 外部 agent 驱动单局流程
  - 状态变化回调驱动动作决策

### 可复用结论
不能直接用于 STS2，但非常适合作为“STS2 通信桥”的设计参考。

---

## 2. `xaved88/bottled_ai`
- 价值定位：**决策层模板**
- 核心能力：
  - 不是大模型，而是手工策略 + 图遍历 + 战斗模拟
  - 战斗中枚举可能出牌顺序
  - 用比较器/评分器判断最优结果
  - 用 handler + strategy 结构做牌、遗物、升级、奖励等选择

### 最值得学习的部分
- **战斗搜索器**：对可能出牌顺序进行模拟
- **结果比较器**：不是只看伤害，而是综合几十项权重
- **handler 机制**：把奖励选择、事件处理、商店购买拆成独立逻辑模块
- **能力边界文档化**：明确缺哪些卡、哪些机制、什么情况下会降级

### 可复用结论
这是最值得复刻到 STS2 的部分。真正的“强度”不会来自 LLM，而会来自这个层级的规则化决策系统。

---

## 3. `ifree/MCPTheSpire`
- 价值定位：**agent 工具层模板**
- 核心能力：
  - 把 Slay the Spire 封装成 MCP 工具
  - 暴露 `get_game_state`、`get_screen_state`、`execute_actions` 等接口
  - 支持安全的读写分离和线程安全执行

### 最值得学习的部分
- **工具化接口设计**
- **状态查询与动作执行分离**
- **批量动作执行**
- **面向 agent 的 API，而不是面向 bot 的耦合式代码**

### 可复用结论
如果目标是让 OpenClaw / MCP client / LLM 都能接入 STS2，这个项目的设计思路非常值得直接借鉴。

---

## 4. `jiegec/STS2FirstMod`
- 价值定位：**STS2 接入入口证明**
- 核心能力：
  - 证明 STS2 已经可以通过 mod 扩展
  - 显示 STS2 当前 mod 开发路径是 `Godot + .NET + sts2.dll`
  - 提供了最小可运行 mod 的工程构建路径

### 可复用结论
这不是 agent 项目，但它解决了最关键的问题：
**STS2 是可以被 mod 接入的。**

---

## 核心判断

## 能不能做出“相等”版本？
可以。

原因：
- STS2 已经有 mod 入口
- STS1 已经有成熟通信层、决策层、MCP 接口层可参考
- 游戏类型仍然是非常适合规则驱动 agent 的卡牌 roguelike

## 能不能做出“优化”版本？
有机会，而且我认为应该直接瞄准“优化版”，而不是纯复刻。

原因：
- STS1 的 bot 多数是历史演进产物，工程结构不一定最佳
- 现在可以直接按“mod 接入层 + agent 工具层 + 策略层”三层重做
- 可以天然为 OpenClaw / MCP / LLM 混合决策留好接口

---

## 推荐架构

## 第一层：STS2 Mod 接入层

职责：直接从游戏内读状态、收动作。

### 需要暴露的数据
- 玩家：
  - HP
  - 最大 HP
  - 格挡
  - 能量
  - 力量/敏捷/易伤/虚弱等状态
- 手牌 / 抽牌堆 / 弃牌堆 / 消耗堆
- 遗物
- 药水 / 可用物品
- 敌人：
  - HP
  - 格挡
  - 意图
  - 状态
- 当前 screen：
  - 战斗
  - 地图
  - 奖励
  - 商店
  - 事件
  - Boss Reward
- 地图节点与可达路线
- 当前可执行动作列表

### 需要暴露的动作
- `play_card`
- `end_turn`
- `choose_reward`
- `choose_path`
- `buy_item`
- `use_potion`
- `select_event_option`
- `start_game`
- `proceed`

### 技术建议
- 用 STS2 mod 直接暴露本地 HTTP / socket / MCP bridge
- 所有 game-changing 操作应在游戏主线程安全执行
- 所有 state read 应该尽可能轻量、结构化、可序列化

---

## 第二层：Agent 工具层

职责：把游戏能力变成外部 agent 可安全调用的接口。

### 推荐接口
- `get_game_state(include=...)`
- `get_screen_state()`
- `get_available_actions()`
- `execute_actions(actions=[])`
- `simulate_action(action)`
- `explain_state()`

### 设计原则
- 尽量不要让 agent 直接从截图推理
- 优先暴露结构化状态
- 批量动作比单动作更高效
- 动作接口应支持索引、名称、ID 三种方式

### 为什么这一层重要
这一层决定以后你能不能：
- 用 OpenClaw 直接玩
- 用 MCP client 接入
- 用规则 bot 和 LLM bot 共用同一套底层能力

---

## 第三层：决策层

建议拆成两个子层。

### A. 规则/搜索决策层
用于：
- 战斗出牌
- 奖励选择
- 升级优先级
- 商店购买
- 路线规划

#### 借鉴 `bottled_ai` 的部分
- battle simulation
- path traversal
- comparator scoring
- handler-based strategy

#### 推荐能力
- 一回合内枚举可能出牌顺序
- 对每种顺序算结果分
- 评分维度至少包括：
  - 击杀收益
  - 受到伤害
  - 资源保留
  - 下回合风险
  - 姿态/球/连锁效果价值

### B. Agent / LLM 决策辅助层
用于：
- 构筑方向解释
- 长期路线判断
- 稀有卡取舍中的高层策略
- 生成可解释决策说明

#### 建议定位
- 不直接替代战斗核心
- 更适合做：
  - 高层策略建议
  - 局外规划
  - 日志说明
  - 人类可读解释

---

## 最现实的开发顺序

## Phase 1 - 接入验证
目标：证明 STS2 能被 agent 安全操控。

### 产出
- 一个 STS2 mod
- 能返回最小 game state
- 能执行最小动作：
  - 出牌
  - 结束回合
  - 选奖励
  - 继续前进

### 成功标准
- 外部程序可以稳定完成一场最简单战斗

---

## Phase 2 - MCP / HTTP 工具化
目标：把 STS2 封装成 agent 工具集。

### 产出
- `get_game_state`
- `get_screen_state`
- `execute_actions`
- 错误处理与线程安全队列

### 成功标准
- OpenClaw 或其他 agent 可以“看状态 -> 下动作 -> 获得反馈”形成闭环

---

## Phase 3 - 规则战斗 AI
目标：做出能稳定打基础战斗的 agent。

### 产出
- 卡牌效果模拟框架
- 出牌顺序搜索器
- 战斗结果评分器

### 成功标准
- 能稳定完成基础怪战斗
- 能在若干角色/基础卡组上跑通

---

## Phase 4 - 局外策略
目标：让 agent 不只是“会打一回合”，而是会打一整局。

### 产出
- 奖励选择器
- 升级逻辑
- 商店购买逻辑
- 路线规划逻辑
- 事件处理逻辑

### 成功标准
- 能自动跑完整局
- 虽然不一定强，但能稳定完成闭环

---

## Phase 5 - 混合优化
目标：做出比传统 bot 更适合 OpenClaw 的版本。

### 方向
- 规则引擎负责稳定执行
- LLM 负责解释和高层规划
- 自动记录对局日志和失败原因
- 通过 growth-log / docs 沉淀策略演化过程

---

## 风险点

## 1. STS2 mod API 还不稳定
STS2 还早期，mod API 可能经常变。

### 应对
- 接入层尽量薄
- 统一封装适配器
- 不要把大量决策逻辑写死在 mod 内部

## 2. 游戏状态可能不如 STS1 那样容易读全
### 应对
- 先做最小可玩状态
- 逐步补细节
- 优先保证战斗状态可用

## 3. 决策空间很大，纯 LLM 不稳定
### 应对
- 战斗必须优先规则化
- LLM 只做辅助层

## 4. 模拟器成本高
### 应对
- 先做高价值卡牌和常见机制
- 和 `bottled_ai` 一样明确记录 coverage 边界

---

## 相比现有项目的优化方向

如果要做“比老项目更好”，我建议重点做这几个优化：

### 1. Agent-native 接口优先
不是只为一个 bot 写，而是为 OpenClaw / MCP / 规则引擎共用而写。

### 2. 决策解释能力
每次选择都能输出：
- 为什么打这张牌
- 为什么选这张卡
- 为什么走这条路线

### 3. 更好的日志与复盘
把失败和策略演化直接沉淀到：
- `growth-log/`
- `docs/`
- GitHub commit history

### 4. 模块解耦
- 游戏接入层
- 工具接口层
- 策略层
- 评估层
分开，后续替换方便

---

## 结论

当前并没有一个现成的 GitHub 项目，能直接让 OpenClaw 自主游玩《杀戮尖塔2》并达到成熟可用。

但从现有项目拼图来看：
- STS1 已经有成熟通信层和策略层
- STS2 已经有 mod 接入入口
- MCP 化方案已经出现

所以最合理的路线不是“等别人做完”，而是：
- 借 `STS2FirstMod` 做接入
- 借 `spirecomm` 做桥接
- 借 `bottled_ai` 做规则决策
- 借 `MCPTheSpire` 做 agent 工具接口

这条路线不只是可做，而且有机会做出一个对 OpenClaw 更友好的优化版实现。

---

## 下一步建议

### 建议立刻做的事
1. 建一个 `sts2-agent` 子目录或独立仓库
2. 先验证 STS2 mod 能否输出最小战斗状态
3. 定义 MCP / HTTP 工具接口草案
4. 先实现最小动作闭环：读手牌 -> 出一张牌 -> 结束回合

### 建议第二步做的事
1. 做战斗状态结构化
2. 做 battle simulator 原型
3. 把最基础角色/卡牌跑通

### 建议第三步做的事
1. 接 OpenClaw
2. 用 growth log 记录策略演化
3. 在 GitHub 上逐步形成可复用框架
